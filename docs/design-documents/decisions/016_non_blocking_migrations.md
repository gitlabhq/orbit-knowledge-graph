## Date

2026-06-04

## Context

The indexer evolves its graph schema by bumping a single global `u32`
(`config/SCHEMA_VERSION`, currently `54`, embedded via `include_str!` in
`crates/indexer/src/schema/version.rs:40`). That one number is overloaded to mean
three different things at once:

1. **Table identity.** Every physical table is prefixed with the version:
   `v54_gl_user`, `v54_gl_edge`, `v54_checkpoint`. The prefix is computed by
   `table_prefix()` (`crates/indexer/src/schema/version.rs:136`) and applied to the
   in-memory ontology by `Ontology::with_schema_version_prefix()`
   (`crates/ontology/src/lib.rs:508`).
2. **Migration trigger.** A binary whose embedded version is greater than the
   active row in `gkg_schema_version` runs `schema::migration::run_if_needed()` at
   dispatcher boot (`crates/indexer/src/schema/migration.rs:74`). This creates a
   **brand-new, empty, prefixed table set** for the whole graph.
3. **Re-index scope.** Because the new `v55_checkpoint` table starts empty, the
   normal poll cycle re-pulls **the entire graph** from epoch. Promotion is gated
   SDLC-only on checkpoint *presence* (`sdlc_count >= enabled_count`,
   `crates/indexer/src/schema/completion.rs`), after which the old prefixed tables
   are dropped past a retention window.

This produces three problems we want to eliminate:

- **Every change is a full re-index.** A one-column ontology addition rebuilds all
  ~100 tables and re-indexes the whole graph, because table identity and re-index
  scope are both bound to the global version.
- **No notion of a delta.** `config/graph.sql` is a regenerated full-snapshot
  artifact (`-- [AUTOGENERATED]`). There is no `up`/`down` step that describes only
  what changed; the migration *is* "throw away the old tables."
- **A correctness hazard.** Promotion is presence-based, while the watermark is
  one-way (`col > last AND col <= current`,
  `crates/indexer/src/modules/sdlc/handler/entity.rs:127`). This is the v48 SDLC
  data-loss incident: a namespace that committed an in-progress checkpoint counted
  toward promotion, the version was promoted and the old tables were dropped, and
  the gap below the watermark became permanently unreachable because incremental
  never re-pulls below itself.

We want migrations that (a) live outside the version-swap machinery, (b) are owned
by the dispatcher/scheduler, (c) drop the incrementing table prefix, (d) apply
changes **in place** and re-index **only the entities that actually changed**, and
(e) keep incremental indexing flowing throughout, as close to free as possible.

## Decision

Replace the global-version table swap with **ordered, additive, in-place
migrations derived from the ontology**, plus **entity-level re-index** driven by
per-entity schema fingerprints. Tables keep stable, unprefixed names for their
entire lifetime.

### Migrations are additive-only

The automated runner supports **only additive changes**, which ClickHouse can
apply in place as metadata operations:

- `ADD COLUMN` (nullable or with `DEFAULT`)
- `ADD INDEX`, `ADD PROJECTION`
- `CREATE TABLE` (a new node or edge table)
- a new `relationship_kind` value into an existing edge table (no DDL — it is a
  string column in `gl_edge`)

Breaking changes — `ORDER BY`/sort-key changes, incompatible `MODIFY COLUMN`,
column or edge-kind renames, drops that live queries depend on — are **explicitly
out of scope for the runner**. ClickHouse cannot apply them in place without a
table rewrite, and an in-place rewrite cannot preserve zero-downtime correctness
the way the per-version swap did. A breaking change is handled as a deliberate,
out-of-band operation that establishes a new baseline (see _Why not the
alternatives_). A build-time lint rejects any generated migration whose `up.sql`
contains a non-additive statement, so "additive-only" is mechanically enforced
rather than a convention.

### Migrations are generated from the ontology as up/down files

`config/graph.sql` (regenerate-the-world) is replaced by an ordered migration set:

```text
config/migrations/
  0001_initial/            # full baseline: today's graph.sql, unprefixed
    up.sql  down.sql  meta.yaml
  0002_add_user_bio/
    up.sql   -> ALTER TABLE gl_user ADD COLUMN bio String DEFAULT '' CODEC(ZSTD(1))
    down.sql -> ALTER TABLE gl_user DROP COLUMN bio
    meta.yaml
```

`meta.yaml` carries what the runner needs beyond raw SQL:

```yaml
touches_tables: [gl_user]
reindex_entities: [User]      # entities whose stored shape changed
class: additive
ontology_fingerprint: <hash>  # ontology state this migration brings us to
```

`mise schema:generate:migration` **diffs the working ontology against the last
applied ontology snapshot** and emits the migration. Humans still edit only YAML
under `config/ontology/`; the migration is a generated, reviewable, committed
artifact. A `build.rs` drift check (the build-time-validation pattern already used
by `crates/gkg-analytics/build.rs`) asserts that **replaying all migrations from
scratch reproduces the ontology's DDL**, so migrations can never silently diverge
from the YAML, and the check fails locally even when CI egress is down.

### Per-entity fingerprint is the unit of "what changed"

Today there is exactly one global version and no per-entity fingerprint. We add a
fingerprint per node type and per edge type:

```text
fingerprint(entity) = hash(columns, types, sort_key, ETL source mapping, transforms)
```

recorded in a control table:

```sql
gkg_entity_schema(entity_type, fingerprint, reindex_status, applied_at)
```

A changed fingerprint means that entity — and only that entity — needs re-index.
This is what turns "re-index only what changed, at the entity level" from a wish
into a mechanical decision.

### Entity-level re-index is a selective checkpoint reset

Checkpoints collapse to a **single, unprefixed `checkpoint` table**. Keys keep
their existing `{scope}.{entity}` shape (`global.User`, `ns.100.MergeRequest`; see
ADR 014). To re-index entity `X`, the runner **tombstones only the `*.X` checkpoint
keys**. The existing per-`{scope}.{entity}` poll cycle then re-pulls `X` from epoch
— and nothing else.

Not every additive migration re-indexes. `reindex_entities` is computed from
whether the change affects extracted/stored values:

- `ADD COLUMN` that an ETL mapping populates → re-index that entity (existing rows
  hold the `DEFAULT` until the backfill catches up; eventual consistency, no
  downtime).
- `ADD INDEX` / `ADD PROJECTION` on existing columns → no re-index; materialize on
  existing parts, new parts get it automatically.
- `CREATE TABLE` for a new entity → no reset needed; the empty checkpoint already
  drives a first full index through the normal path.

### Incremental indexing stays free

Because every entity keeps its own checkpoint, **incremental indexing for unchanged
entities never pauses** — that is the property we get for free by no longer wiping
the checkpoint table on every schema change. The entity under re-index also keeps
receiving new data: the watermark filter's `current` bound is dispatch-time
wall-clock, and the cursor state machine (`cursor_values: Some` → resume
pagination, `None` → completed → advance watermark,
`crates/indexer/src/modules/sdlc/handler/entity.rs:127`) already resumes a long
backfill across dispatches and then converges to live. A re-indexing entity needs
no separate incremental path; it catches up to the present on its own.

### The dispatcher/scheduler owns migrations

Everything moves under the existing scheduler (`crates/indexer/src/scheduler/mod.rs`,
`ScheduledTask`):

- **`MigrationRunner`** (boot step + scheduled task, under the existing
  `indexing_locks/schema_migration` NATS KV lock) reads the embedded migration set,
  diffs it against a `gkg_migrations` ledger, applies unapplied `up` migrations in
  order **in place**, then tombstones the checkpoint keys for each migration's
  `reindex_entities`.
- The normal dispatch loops (`NamespaceDispatcher`,
  `NamespaceCodeBackfillDispatcher`, …) are unchanged — empty/tombstoned checkpoints
  already drive the re-pull. The `campaign` tag can be repurposed to label a
  per-migration re-index wave for observability, or retired.
- The presence-based promotion gate in `MigrationCompletionChecker` is **removed**.
  With in-place additive migrations there is no swap to promote: an incomplete
  re-index is temporary staleness in a live table, not permanent loss. The task is
  reduced to observing per-entity re-index progress for metrics.

### Control tables

```sql
gkg_migrations(name, checksum, class, applied_at, status)          -- the ledger
gkg_entity_schema(entity_type, fingerprint, reindex_status, applied_at)
checkpoint(key, watermark, cursor_values, _version, _deleted)      -- unprefixed
```

`gkg_schema_version`, `table_prefix()`, and `with_schema_version_prefix()` are
deleted at the end of the migration path. The webserver readiness gate
(`crates/gkg-server/src/schema_watcher.rs`) changes from "is `active_version ==
embedded_version`" to "are all migrations my binary expects present in
`gkg_migrations`".

### Cutover from the v54_ prefix: one fresh rebuild

The move from `v54_`-prefixed tables to stable canonical names is a one-time **fresh
rebuild**: `0001_initial` creates the unprefixed canonical tables, the graph is
indexed into them once, and the `v54_` table set is dropped afterward. We pay one
full re-index at cutover in exchange for never paying a forced full re-index again,
and for a clean slate with no prefix-rename edge cases.

## Incremental path

Each stage is independently shippable and reversible.

1. **Foundations (no behavior change).** Add `gkg_migrations` and
   `gkg_entity_schema` control tables and per-entity fingerprint computation.
   Shadow-write them alongside the live system; assert in tests that fingerprints
   reconcile with reality. Nothing reads them yet.
2. **Migration generator + build-time drift check.** `mise schema:generate:migration`
   diffs the ontology and emits `up.sql`/`down.sql`/`meta.yaml`; `0001_initial`
   equals today's `graph.sql`. `build.rs` asserts replay == ontology DDL, and the
   additive-only lint lands. `graph.sql` is still generated in parallel as a safety
   net.
3. **MigrationRunner against unprefixed tables (dev/shadow only).** Prove "apply all
   migrations from scratch == current schema" in integration tests. No production
   cutover.
4. **Selective checkpoint reset.** Implement `*.X` tombstoning and per-entity
   re-index dispatch. Verify unchanged entities' incremental flow is untouched.
   Testable before the prefix is dropped.
5. **Cutover: fresh rebuild into canonical names.** Index once into unprefixed
   tables, flip reads/writes off the prefix, switch the readiness gate to the
   migrations ledger, drop the `v54_` set.
6. **Remove the old system.** Delete the `gkg_schema_version` migration path, the
   presence-based promotion gate, `table_prefix`/`with_schema_version_prefix`, and
   retire `graph.sql` as the source-of-truth artifact.

## Why not the alternatives

### Keep the global version and prefixed-table swap

This is the status quo. It guarantees a fully-built new table set before cutover,
but binds table identity and re-index scope to one number, so any change rebuilds
the whole graph, and the presence-based promotion gate is the direct cause of the
v48 data loss. The whole point of this ADR is to decouple those.

### Per-entity shadow table + atomic RENAME for breaking changes

Considered and rejected for now. It would preserve zero-downtime for sort-key/type
rewrites (build `gl_user__next`, re-index that one entity, atomic `RENAME`), by
replacing the global prefix with a per-entity physical-table pointer resolved at
query-compile time. It is strictly more machinery — a per-entity pointer, a
per-table completion gate, and query-layer indirection — to support changes that
are rare. We chose **additive-only** instead: breaking changes are handled
out-of-band as a new baseline, keeping the runner small and the query path direct.
If breaking-change frequency rises, this is the natural extension point.

### Allow non-additive changes in place

ClickHouse cannot change `ORDER BY` or do an incompatible `MODIFY COLUMN` without
rewriting data, and an in-place rewrite on a live table cannot preserve query
correctness during the rewrite. Allowing it would smuggle back the exact
swap-during-serve hazard we are removing. Forbidding it (with a lint) is the safe
boundary.

## Consequences

### What improves

- A one-column ontology addition becomes one `ALTER ... ADD COLUMN` plus a re-index
  of one entity, instead of a full-graph rebuild.
- Tables have stable names; the `v54_` prefix and its query-time injection are gone.
- Incremental indexing for unchanged entities never pauses across a schema change.
- The v48 failure mode is structurally impossible for additive migrations: there is
  no promotion gate and no table drop, so an incomplete re-index is recoverable
  staleness, not loss.
- Migrations are reviewable deltas (`up`/`down`) with a build-time guarantee that
  they match the ontology.

### What gets harder

- Breaking schema changes are no longer self-service; they require a deliberate
  out-of-band baseline operation.
- A new artifact set (`config/migrations/`) and generator must be maintained, and
  the additive-only invariant must be enforced by lint + `build.rs`.
- During an additive backfill, queries can read `DEFAULT` values for the new column
  on not-yet-re-indexed rows. This is eventual consistency by design and must be
  documented for query consumers.
- Cutover pays one full re-index.
