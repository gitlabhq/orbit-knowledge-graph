5. Interfaces
=============

Runtime Search Flow and Known Issue
------------------------------------

The code/SDLC search flow in cloud mode involves a synchronous chain through
DAP, Orbit, ClickHouse, and the Rails redaction service.  This chain
**blocks a Puma thread** for the entire duration of the request.

.. uml:: puml/gkg_search_sequence.puml
   :caption: Search flow --- query engine compilation, auth, and redaction all
             block the Puma thread

**The problem:** A search request holds a Puma thread from the moment Rails
receives it until Orbit finishes: DSL compilation (4-phase pipeline),
ClickHouse query, Redis auth cache lookup, and N round-trips to the Rails
redaction service for ``Ability.allowed?`` permission filtering.  At 100x
concurrency this exhausts the Puma pool and degrades all Rails request
handling --- not just search.

**Remediation path:** Move the search request to an async job (Sidekiq) or
adopt a streaming/callback pattern so the Puma thread is released immediately.
Orbit can notify Rails via webhook or SSE when results are ready.

Graph Query Engine --- Five Query Types
----------------------------------------

The JSON DSL supports five query types, all compiled to parameterized
ClickHouse SQL:

.. list-table::
   :widths: 25 75
   :header-rows: 1

   * - Type
     - Description
   * - **Multi-hop traversal**
     - Walk the graph N hops from a starting node, following specified edge
       types.  Edge-centric traversal with SIP optimization.
   * - **Search**
     - Full-text search across node properties (definitions, names, etc.).
   * - **Aggregation**
     - Count, group, and summarize graph data (e.g., "issues by label per
       project").
   * - **Pathfinding**
     - Find paths between two nodes through the graph.
   * - **Nearest neighbors**
     - Find nodes most similar to a given node by graph distance or property
       similarity.
